1. Introduction

1.1 Purpose

This Low Level Design document describes the implementation-level design of the Diabetes Insight system. It explains how the approved HLD modules are translated into concrete Python files, functions, data structures, and runtime interactions. The document supports development, academic evaluation, maintenance, and technical understanding of the application.

The LLD focuses on how the Streamlit user interface, CSV storage, model loading, prediction logic, training workflow, and runtime validation are organized in code. Each major module is mapped to responsibilities, interfaces, local data, global data usage, and error handling behavior.

1.2 Scope

The scope of this LLD includes detailed design of the user login flow, input form handling, prediction pipeline, model training batch process, CSV persistence, module interactions, exception management, and UI process logic. It applies to the current university project implementation in Python and Streamlit and assumes that the model is deployed as a pre-trained joblib file.

2. Conventions

The following conventions are used throughout this document:

  1. Python files and functions are written in monospace, for example app.py and load_model().
  2. Client side refers to UI behavior in the browser rendered by Streamlit.
  3. Server side refers to Python execution logic responsible for validation, inference, and persistence.
  4. The term data store refers to the CSV persistence file users.csv.
  5. Risk thresholds are interpreted as Low, Moderate, and High according to configured probability boundaries.
  6. Session state refers to Streamlit runtime memory stored in st.session_state.

3. Design Constraints And Considerations

The implementation design must comply with the project constraints already approved in the SRS and HLD. The application is lightweight, file-based, and suitable for academic demonstration rather than production medical deployment. The key implementation concern is to keep the code simple, readable, and modular while ensuring correct inference and predictable UI behavior.

5.3 Data Store

The principal data store is users.csv. It is used to persist user information and the most recent or cumulative prediction details, depending on the storage policy implemented in user_store.py. The data store is intentionally simple and optimized for academic demonstration rather than transactional production use.

5.4 Processing Specification

  1. User opens the application and logs in with name and gender.
  2. Application stores login details in st.session_state.
  3. Dashboard is rendered with health input controls.
  4. Pregnancies field is enabled for female users and disabled for male users.
  5. User enters all medical values and presses the predict button.
  6. Input validation is performed for completeness, data type, and range correctness.
  7. Validated values are converted into a one-record dictionary or DataFrame.
  8. The prediction module loads the model using joblib if not already cached.
  9. The feature vector is ordered according to the training schema.
  10. The model generates class prediction and diabetic-class probability.
  11. Probability is converted into Low, Moderate, or High risk classification.
  12. UI displays the label, probability, and interpretation text.
  13. CSV storage module saves or updates the user prediction record.

6.4 Processing Logic

The full application flow is linear and state-controlled. The application starts with login, validates and stores the session, renders the dashboard, collects health values, validates them, loads the trained model, predicts the probability and class, classifies the risk, displays the result, and saves the output to CSV. Repeat prediction is supported within the same user session, while logout clears the state and returns the user to the login page.

6.5 Exception / Error Handling

Error handling is designed to be defensive and user-safe. Invalid input is blocked before prediction, missing model artifacts stop inference immediately, duplicate users can be prevented where required, and storage errors never result in raw internal tracebacks being shown to the user. In all cases, the system attempts to preserve the session and keep the UI responsive.

7. Software Detailed Design For Batch Processes

The main batch task in the system is model training. It is executed offline by the project developer rather than the end user. The task reads the diabetes dataset, applies preprocessing, trains one or more candidate classifiers, evaluates them, and saves the best-performing model artifact for runtime use.

Global data structures include the session dictionary maintained by Streamlit, the ordered list of feature columns expected by the model, the CSV column schema used by the persistence layer, and the model object loaded from the joblib file during inference.

7.7 Call Convention

Modules communicate through standard Python function calls. Input arguments are passed as primitive values, dictionaries, or DataFrames, while returned values are passed as dictionaries, Booleans, status objects, or model instances. Naming and return conventions are kept simple to make the code easy to inspect during academic review.

7.8 Module Specifications

7.8.1 app.py

Description: Main Streamlit controller responsible for page rendering, user flow, event handling, and integration of prediction and storage services.

Interface: Uses Streamlit widgets for input and calls create_user(), predict_single(), and upsert_user_prediction().

File Usage: Indirectly accesses users.csv and the model file via dependent modules.

Global Data Usage: Reads and writes st.session_state for login status, user identity, gender, and latest result.

Local Data: Temporary form values, event flags, user input dictionary, result objects.

Logic: Initialize state, display login, validate identity, render dashboard, capture health input, invoke prediction, display result, support logout.

Error Handling: Displays login warnings, validation errors, duplicate-user warnings, model errors, and storage-save failures.

7.8.2 predict.py

Description: Inference module that encapsulates model loading, feature preparation, prediction execution, and risk classification logic.

Interface: Exposes load_model(), predict_single(), and classify_risk().

File Usage: Reads models/best_model.joblib.

Global Data Usage: Uses configured model path, feature ordering, and threshold constants.

Local Data: One-record DataFrame, predicted label, probability variable, risk class string.

Logic: Verify model exists, load model, structure input, call prediction methods, compute probability, map risk, return result.

Error Handling: Handles missing model, malformed input, incompatible schema, and inference-time exceptions.

7.8.3 preprocessing.py

Description: Defines the training-time data transformation rules and provides preprocessing objects to the training module.

Interface: Exposes preprocessing builder functions and feature configuration helpers.

File Usage: Typically interacts with training data indirectly through train_model.py.

Global Data Usage: Feature lists, numeric or categorical field grouping, preprocessing configuration.

Local Data: Transformers, imputer configuration, scalers, and encoder definitions.

Logic: Identify columns, define numeric transformations, define categorical transformations if required, and build a reusable transformer pipeline.

Error Handling: Raises or reports missing column definitions and transformer construction issues.

7.8.4 train_model.py

Description: Offline batch-training module that reads the source dataset, trains candidate models, compares them, and saves the best model.

Interface: Exposes a training entry point and internal functions for model evaluation and serialization.

File Usage: Reads data/diabetes.csv and writes models/best_model.joblib.

Global Data Usage: Dataset path, target column, feature columns, output model path.

Local Data: Train/test splits, model candidates, evaluation scores, fitted best model.

Logic: Load dataset, split features and target, apply preprocessing, train candidate models, evaluate, choose best, save artifact.

Error Handling: Handles missing dataset, missing target column, training-time errors, and joblib serialization failures.

7.8.5 user_store.py

Description: CSV persistence layer that ensures file existence, manages schema, creates users, detects duplicates, and saves prediction results.

Interface: Exposes ensure_user_store(), create_user(), get_user(), and upsert_user_prediction().

File Usage: Reads and writes data/users.csv.

Global Data Usage: CSV path and ordered schema columns.

Local Data: DataFrame rows, duplicate masks, update masks, row dictionaries.

Logic: Ensure file exists, load rows, check existence, create or update target row, write back to CSV.

Error Handling: Handles file absence, malformed data, duplicate users, and write-permission failures.

9. Alternate Design Methods Considered

Why Streamlit over React: Streamlit was selected because it allows the project to stay fully in Python, which is a strong fit for a machine-learning-centered academic system. A React frontend would require a separate API layer, additional state management, and more deployment complexity. Streamlit provides a faster, clearer, and more maintainable path for university evaluation.

Why CSV over a database: CSV storage satisfies the lightweight data persistence requirement, is easy to inspect during project review, avoids database configuration, and remains sufficient for the expected low data volume. A relational database would improve scalability but would add complexity beyond the needs of the current scope.

Why Random Forest over other algorithms: Random Forest is well suited to tabular medical data, supports probability output, handles nonlinear relationships effectively, and is simple to integrate with scikit-learn pipelines. It offers a practical balance between performance and implementation simplicity for a student AI project.
