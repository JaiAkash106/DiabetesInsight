1. Introduction

1.1 Purpose

The purpose of this High Level Design document is to define a clear architectural model for the implementation of Diabetes Insight, a web-based diabetes risk prediction system built with Streamlit and a pre-trained Random Forest model. This document bridges the gap between the Software Requirements Specification and the actual implementation approach by showing how the major system modules, data flow, user interaction, and supporting components fit together.

The HLD is intended to support university evaluation, technical review, implementation planning, and future enhancement of the application. It provides a design view that is detailed enough for academic assessment while remaining understandable for student developers and project supervisors.

1.2 Scope

The scope of this design covers the complete high-level solution for the Diabetes Insight application, including the Streamlit presentation layer, user login workflow, gender-sensitive form behavior, preprocessing and inference pipeline, result classification logic, CSV-based persistence, and exception handling. The design assumes that the machine learning model has already been trained and stored as a joblib artifact before runtime.

The document does not include low-level code design, algorithm training mathematics, or production-grade deployment architecture. Instead, it focuses on the system-level organization needed for a functional, maintainable, and presentation-ready academic project.

2. System Environment

2.1 System Input

The system accepts two main categories of input:

  • User identification input: name and gender.
  • Medical prediction input: pregnancies, glucose, blood pressure, skin thickness, insulin, BMI, diabetes pedigree function, and age.

The pregnancies field is dynamically controlled by the system. It remains editable for female users and disabled or fixed to zero for male users. All input values are collected using Streamlit widgets and then validated before prediction.

2.2 System Output

The system produces both on-screen and file-based output:

  • On-screen output: validation messages, prediction probability, predicted class, risk level, and interpretation text.
  • File-based output: CSV records containing login details, medical inputs, prediction results, and timestamps.

4. Alternate Design Methods Considered

4.1 Design Methods Analysis

Three implementation strategies were considered during the design stage. The first was a traditional web stack using a Python backend and relational database. This approach offered better scalability and stronger separation between concerns, but it introduced unnecessary complexity for a university mini-project.

The second was a desktop GUI built with a native Python toolkit. While simple to run locally, it was less suitable for demonstrating a modern AI web workflow. The final option was a Streamlit-based solution with file-backed storage. This option was chosen because it aligns directly with the project requirements, reduces infrastructure overhead, and supports rapid integration with machine learning components.

4.2 Basis on Which the Design Is Arrived

The chosen design was evaluated using the following criteria:

  • simplicity of development and deployment,
  • compatibility with Python-based machine learning workflows,
  • ease of demonstration during university review,
  • maintainability by student developers,
  • low cost and minimal dependency on external infrastructure.
Selected approach: Streamlit + pre-trained Random Forest model + CSV persistence. This option provides the best balance between academic value, ease of explanation, and implementation speed.

The implementation may remain function-oriented, but the logical design can still be described with object-oriented responsibilities. An AppController coordinates user flow, a SessionManager holds login context, an InputValidator checks the form, a PredictionService handles model loading and inference, a RiskClassifier maps the probability to risk labels, and a CSVRepository manages file persistence. 6.2 Entity, Attribute and Their Relationships

The logical data model contains three principal entities. The User entity holds simple login information such as name and gender. The Prediction Record entity stores the health input vector, the predicted class, the probability score, the derived risk level, and the timestamp of each prediction. The Model Metadata entity represents information about the currently loaded model artifact and is useful for traceability during academic review.

6.3 Data Sizing and Performance

CSV storage is intentionally lightweight. The design assumes small to moderate academic usage, typically from tens to a few thousand rows. At this scale, append operations remain efficient, and the file size is still manageable without indexing or database optimization. Model inference occurs for a single record at a time, so latency remains low on a standard student laptop.

6.4 Processing Specification

  1. User login data is captured and stored in session state.
  2. Medical input is collected through the web form and validated.
  3. Gender-based pregnancies logic is applied before model inference.
  4. The feature vector is arranged in the exact order expected by the trained model.
  5. The model is loaded using joblib and used to compute the class and probability.
  6. The probability is classified as Low, Moderate, or High risk.
  7. The final result is displayed and appended to the CSV record store.

7. Prototype

7.1 Scope

The prototype covers the complete user-facing flow required for the project: login, medical data entry, prediction, result display, and record storage. It demonstrates the interaction between the Streamlit UI and the trained Random Forest model in a way that is suitable for university evaluation.

7.2 Objective

The objective of the prototype is to validate usability, confirm successful integration of the machine learning pipeline, and show that the system can guide a user from raw health input to an understandable risk assessment. It also demonstrates the use of a lightweight persistence mechanism without requiring a database.

7.3 Observations

The prototype supports a clean and linear workflow. The conditional pregnancies control improves correctness, model integration remains straightforward due to consistent feature mapping, and the UI is simple enough for non-technical project evaluators to understand during demonstration.

7.4 Conclusion

The prototype is sufficient for academic demonstration and clearly shows how AI prediction, web interface design, and lightweight data persistence can be combined in a practical healthcare-themed student project.

10. Backup and Recovery Procedures

Because the system relies on a model artifact and CSV files rather than a database, backup and recovery are file-centered. The recommended procedure is to maintain a protected backup copy of the joblib model and a periodic backup of CSV records. If the model file becomes unavailable, the application should not attempt prediction until the correct artifact is restored. If CSV files are missing, the system may recreate the file structure, but historical records can only be recovered from backup copies.

11. Design Constraints

  • The application must be built with Streamlit.
  • The model must be pre-trained and loaded with joblib at runtime.
  • Persistent storage must be CSV-based and not database-driven.
  • The login process must remain simple and rely on name plus gender.
  • The pregnancies field must be disabled for male users.
  • The project is intended for educational use and not clinical deployment.

12. Design Criteria

The design is guided by the following criteria:

  • Simplicity: the system should be easy to explain and demonstrate.
  • Functional completeness: all required SRS features must be reflected in the design.
  • Modularity: UI, validation, inference, and storage should remain separable.
  • Maintainability: future updates such as new thresholds or fields should be straightforward.
  • Responsiveness: prediction should complete fast enough for interactive use.

13. Reliability Requirements

The high-level design supports reliability through validation-before-inference, deterministic feature ordering, safe model loading, and controlled persistence behavior. The same valid inputs should produce the same outputs as long as the deployed model version remains unchanged. Error conditions must be surfaced as understandable messages rather than silent failures or application crashes.

14. Anticipated Changes

  • migration from CSV storage to a relational database,
  • support for stronger user authentication,
  • multiple model comparison and explainability features,
  • additional charts, dashboards, and downloadable reports,
  • cloud deployment and multi-user access control.
